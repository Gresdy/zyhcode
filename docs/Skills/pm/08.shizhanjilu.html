产品设计 Skills 使用指南

zyh2026-04-02

产品经理 Skills 使用指南

本文档记录使用 brainstorming → write-a-prd → writing-plans 三 Skill 组合完成"千里眼智能感知系统"产品设计的完整流程与使用体验。

一、组合使用价值

这三个 Skill 形成了一套完整的从想法到可执行计划的产品工作流：

阶段Skill核心价值产出物
构思阶段01 brainstorming发散思路、收敛方案、识别风险架构设计文档
需求阶段02 write-a-prd深度访谈、结构化输出、逻辑验证PRD 文档
执行阶段03 writing-plans细粒度拆解、验收标准、任务清单可执行实施计划

核心理念：把复杂目标拆解成带验收标准的微任务清单，适用于产品经理规划项目执行路径、制定内容发布计划，或在与研发对齐时提供细粒度任务分解参考。

二、千里眼系统 Demo 背景

2.1 项目简介

千里眼系统是一款基于遥感、摄像头、各种传感器的智能感知平台，融合 Agent 技术，实现任务的自动规划、自动执行、自动调度。用户可通过自然语言发起任务，系统自动调用卫星、无人机、摄像头等感知设备，自动安排人员线下巡查，形成完整的"感知-分析-决策-执行-反馈"闭环。

2.2 核心模块

模块说明
感知中心整合卫星、无人机、摄像头、传感器数据
模型中心统一管理 LLM、多模态模型的接入与路由
智能体中心协调/规划/执行/监控四类 Agent 协作
任务中心统一管理任务创建、流转、追踪
场景中心可视化编排业务场景
调度执行中心统一调度人员、设备、第三方系统

三、完整使用流程

3.1 第一步：01 brainstorming（方案头脑风暴）

使用场景：项目初期，需求模糊、方向不确定

对话式工作方式：

  • 一次只问一个问题，像资深产品顾问一样逐步厘清需求
  • 主动提出 2-3 个可行方案并给出有理由的推荐
  • 逐步收敛，最终产出包含架构、数据流、测试策略的完整设计文档

在千里眼项目中的应用：

当我们构思"如何让用户通过自然语言完成复杂的感知任务"时，brainstorming 帮助我们：

  1. 明确核心问题：用户不是专业人员，如何降低使用门槛？
  2. 探索解决方案：
    • 方案 A：预设固定模板，用户填空
    • 方案 B：意图识别 + 参数补全
    • 方案 C：多轮对话式引导
  3. 确定架构方向：选择多 Agent 协作架构（协调 Agent 理解需求，规划 Agent 拆解任务，执行 Agent 调用工具）

使用感受：适合"我有一个方向但不知道怎么落地"的场景，通过引导式提问帮助团队理清思路。

3.2 第二步：02 write-a-prd（需求设计）

使用场景：方案确认后，输出结构化、可执行的 PRD

访谈式工作方式：

  • 先让你详细描述问题和方案想法
  • 对每个设计分支追问到底，直到达成共同理解
  • 最终生成一份可直接作为 GitHub Issue 提交的 PRD 文档

PRD 文档结构：

1. 从用户视角描述面临的问题
2. 从用户视角描述解决方案
3. 列举详细的用户故事类型
4. 模块划分、接口设计、架构等信息
5. 测试范围、测试策略推荐
6. 其他需要补充的内容

在千里眼项目中的应用：

基于 brainstorming 确定的架构，write-a-prd 帮助我们将"感知中心"模块的需求细化：

  • 问题定义：政府巡查人员需要花费大量时间现场勘查，效率低下
  • 解决方案：用户发起自然语言任务 → Agent 自动规划 → 调度感知设备 → 汇总分析结果
  • 用户故事：
    • 作为巡查人员，我想通过语音说"检查 XX 区域是否有新增建筑"，系统自动完成卫星影像对比并返回结果
    • 作为管理员，我想配置哪些任务需要人工审核，避免高成本误操作
  • 边界 Case：设备离线如何处理？模型调用失败是否降级？

使用感受：访谈式追问能提前暴露逻辑漏洞和边界遗漏，避免开发到一半才发现需求不清。

3.3 第三步：03 writing-plans（实施计划拆分）

使用场景：PRD 确认后，生成细粒度可执行任务

工作方式：

  • 将复杂需求拆解为极细颗粒度的执行计划
  • 每个步骤仅需 2-5 分钟
  • 包含明确的文件路径、命令和验收预期

计划文档结构：

- 功能目标（一句话）
- 架构说明（2-3 句）
- 技术栈
- 文件结构 Map
- 每个任务块的分步骤检查清单（含代码、命令、预期输出）

在千里眼项目中的应用：

将"感知中心-摄像头接入"模块拆解为可执行任务：

任务 1：搭建摄像头接入框架
├── 步骤 1.1：创建 adapter 目录结构
│   └── 验收：目录存在，包含 base.py 和 rtsp.py
├── 步骤 1.2：实现基础 Adapter 基类
│   └── 验收：base.py 包含 connect(), disconnect(), get_frame() 抽象方法
└── 步骤 1.3：实现 RTSP 协议适配器
    └── 验收：能够获取指定摄像头的实时帧

使用感受：与研发对齐时，提供开发视角的实施计划草稿，让研发评估工作量更准确。

五、使用技巧与注意事项

5.1 brainstorming 使用技巧

  • 先说背景再说目标：让 AI 理解你的业务上下文
  • 接受方案发散：不要急于否定某个方案，先听完推荐理由
  • 记录追问：过程中发现的新问题可以先记下来，不打断当前思路

5.2 write-a-prd 使用技巧

  • 准备具体案例：带上实际用户场景比抽象描述更有效
  • 主动补充边界 Case：等 AI 追问比较被动，自己先想清楚
  • 善用 GitHub Issue 格式：输出的 PRD 可以直接提交评审

5.3 writing-plans 使用技巧

  • 与研发一起看：计划产出后拉着研发过一遍，确保可行性
  • 验收标准要具体：不要写"功能正常"，要写"输入 XX 返回 YY"
  • 拆分粒度要适中：2-5 分钟一个任务，避免过于碎片化

六、适用场景总结

场景推荐组合
新功能概念评审，在写 PRD 前发散思路01 → 02
业务流程再设计，梳理复杂逻辑01
PRD 撰写，确保需求无遗漏02
与研发对齐，评估工作量02 → 03
制定项目执行计划03
配合 brainstorming 使用，方案确认后自动衔接生成实施计划01 → 03

七、实际 Demo 示例

以下是我们使用这三个 Skills 完成"千里眼-感知中心"模块的完整过程：

7.1 brainstorming 阶段

用户输入：

我想做一个感知中心，整合卫星、无人机、摄像头、传感器的数据。用户可以通过自然语言发起任务，系统自动调度这些设备完成感知工作。

Skill 输出（关键部分）：

  • 明确了感知中心的定位是"数据基础"
  • 确定了数据处理流程：采集 → 格式转换 → 预处理 → AI分析 → 结果输出
  • 提出了感知中心与模型中心、智能体中心的交互关系

7.2 write-a-prd 阶段

用户输入：

基于之前的设计，为"摄像头接入模块"撰写 PRD。需要支持 RTSP 和 GB28181 协议。

Skill 输出（关键部分）：

  • 用户故事：作为租户管理员，我想接入自己的摄像头，以便实时监控指定区域
  • 接口设计：统一 /api/v1/camera/devices 端点
  • 异常处理：设备离线时返回 503 并触发告警
  • 测试策略：Mock RTSP 流进行单元测试

7.3 writing-plans 阶段

用户输入：

把"摄像头接入模块"的 PRD 拆分成可执行任务

Skill 输出（关键部分）：

摄像头接入模块实施计划

功能目标：实现标准化的摄像头设备接入，支持 RTSP 和 GB28181 协议

架构说明：
- 感知中心模块依赖 common 层数据库和消息队列
- 摄像头适配器采用策略模式，支持多协议扩展

技术栈：FastAPI + OpenCV + ffmpeg

文件结构：
perception/src/adapters/camera/
├── __init__.py
├── base.py      # 基类定义
├── rtsp.py      # RTSP 协议实现
└── gb28181.py   # GB28181 协议实现

任务清单：

[ 任务 1 ] 创建摄像头适配器基础架构
  [ 1.1 ] 创建目录结构
    命令：mkdir -p perception/src/adapters/camera
    验收：目录存在且包含 __init__.py

  [ 1.2 ] 实现 CameraAdapter 基类
    文件：perception/src/adapters/camera/base.py
    验收：基类包含 connect(), disconnect(), get_frame() 抽象方法

  [ 1.3 ] 实现 CameraDevice 模型
    文件：perception/src/adapters/camera/models.py
    验收：包含 device_id, name, address, port, protocol, status 字段

[ 任务 2 ] 实现 RTSP 协议适配器
  [ 2.1 ] 实现 RTSP 连接逻辑
    命令：pytest tests/adapters/camera/test_rtsp.py
    验收：测试通过，能够获取指定摄像头的实时帧

  [ 2.2 ] 实现帧获取方法
    验收：get_frame() 返回 numpy.ndarray 格式图像

[ 任务 3 ] 实现 GB28181 协议适配器
  ...

[ 任务 4 ] 集成到感知中心 API
  [ 4.1 ] 注册路由
    文件：perception/src/api/camera.py
    验收：GET /api/v1/camera/devices 返回设备列表

八、结论

通过 brainstorming → write-a-prd → writing-plans 的组合使用，我们完成了千里眼智能感知系统的完整产品设计流程：

  1. brainstorming 帮助在需求模糊期快速厘清方向，避免走弯路
  2. write-a-prd 确保需求结构完整、逻辑严密，经得起评审
  3. writing-plans 产出可执行的任务清单，让开发团队能够准确评估工时

这套组合特别适合复杂系统设计和跨团队协作场景，能够显著提升产品需求的清晰度和可执行性。
